# UUID Column Addition Plan ‚Äî `model_cards` and `datasheets`

## Context

Following the recent thread with Beth (PI) and William on persistent identifiers, the team agreed to add a UUID column to `model_cards` and `datasheets` as the external-facing PID. The existing `bigint` primary keys stay as the internal identifiers; UUID is an additive, reversible change with **no FK migration across child tables**.

**Motivating incident.** The Neo4j ‚Üí Postgres migration earlier this year included an accidental overwrite that required manual re-entry of model cards. Two takeaways:

- Postgres `bigint GENERATED BY DEFAULT AS IDENTITY` IDs are already permanent at the DB layer (monotonic, immutable, WAL-durable).
- The real protection against migration-style incidents is backup discipline, not schema choice. UUIDs make recovery easier (re-import preserves identifiers) and give consumers a stable external handle aligned with Beth's PID guardrails: external, semantically neutral, 1:1, immutable, indexed, resolvable.

Beth explicitly called out that a bare UUID is not a fully-resolvable PID ‚Äî a resolver (e.g. `patra.pods.icicleai.tapis.io/modelcard/<uuid>` or DOI / Handle / ARK registration) is needed for that. **Resolvability is deliberately deferred to a separate RFC** so this change stays focused.

**Historical context.** The Neo4j iteration (`patra-knowledge-base-dev/ingester/neo4j_ingester.py:38`) used `str(uuid.uuid4())` for model card IDs, stored as the `external_id` property on the `ModelCard` node. So v4 is consistent with the original design ‚Äî this change restores a property that was dropped during the Postgres migration.

## Locked-in decisions

| # | Decision | Choice |
|---|---|---|
| 1 | UUID type | Native Postgres `uuid` (16 bytes, pgcrypto-backed) |
| 2 | UUID version | v4 random via `gen_random_uuid()` |
| 3 | Column name | `uuid` (symmetric on both `model_cards` and `datasheets`) |
| 4 | Constraints | `NOT NULL`, `UNIQUE`, `DEFAULT gen_random_uuid()` |
| 5 | Relationship to bigint PK | Added alongside; bigint stays the PK and FK target. **Zero FK refactor across child tables.** |
| 6 | Semantic neutrality | v4 random (not v5 content-hash) ‚Äî aligns with Beth's guardrail |
| 7 | Duplicate prevention | Application-level `_find_duplicate_model_card` / `_find_duplicate_datasheet` continue to handle content-equality dedup; UUID's `UNIQUE` constraint is a defense-in-depth layer |
| 8 | Column order | Immediately after the PK column |
| 9 | Scope of this change | DBML + `db/bootstrap_schema.sql` + minimal REST-server wiring to return UUID in create + detail responses. `external_id` (bigint) stays for backward compatibility |
| 10 | Resolver | Deferred ‚Äî separate RFC |

## Pre-migration checks (before any `ALTER TABLE` against production `patradb`)

1. **Confirm live schema state.** Query production `patradb` to see whether the dormant `model_cards.patra_mc_id uuid` column exists and is populated:

   ```sql
   SELECT column_name, data_type, is_nullable
   FROM information_schema.columns
   WHERE table_name = 'model_cards' AND column_name IN ('uuid', 'patra_mc_id');

   -- if the column exists:
   SELECT COUNT(*), COUNT(patra_mc_id) FROM model_cards;
   ```

   If `patra_mc_id` exists and is populated, **adopt those UUIDs** (rename to `uuid` or migrate values) rather than generating fresh ones. If it exists but is empty, drop it before adding `uuid` to avoid column confusion.

2. **Backup and dev clone.** Before running any schema change against live `patradb`:

   - Take a full `pg_dump --format=custom` backup via DBeaver's Backup wizard. Move the `.dump` file to durable off-pod storage (Tapis Files / S3 / team backup volume).
   - Verify restore works against a throwaway Docker Postgres (counts match).
   - Create a `patradev` clone (e.g. `CREATE DATABASE patradev WITH TEMPLATE patradb`) and apply the migration there first.

## Changes

### A. DBML files *(already landed)*

`db/schema.dbml`, `db/schema_model_cards.dbml`, `db/schema_datasheets.dbml` already have:

```dbml
uuid uuid [not null, unique, default: `gen_random_uuid()`]
```

immediately after `id` on `model_cards` and after `identifier` on `datasheets`. No further DBML edits in this PR.

### B. `db/bootstrap_schema.sql`

Append idempotent `ALTER TABLE` statements + the `pgcrypto` extension declaration:

```sql
-- UUID external identifiers (external PID, complements the bigint internal id).
-- See poc/UUID_COLUMN_PLAN.md for the design rationale.

CREATE EXTENSION IF NOT EXISTS pgcrypto;

ALTER TABLE model_cards
  ADD COLUMN IF NOT EXISTS uuid uuid NOT NULL DEFAULT gen_random_uuid();

DO $$
BEGIN
  IF NOT EXISTS (SELECT 1 FROM pg_constraint WHERE conname = 'model_cards_uuid_key') THEN
    ALTER TABLE model_cards ADD CONSTRAINT model_cards_uuid_key UNIQUE (uuid);
  END IF;
END $$;

ALTER TABLE datasheets
  ADD COLUMN IF NOT EXISTS uuid uuid NOT NULL DEFAULT gen_random_uuid();

DO $$
BEGIN
  IF NOT EXISTS (SELECT 1 FROM pg_constraint WHERE conname = 'datasheets_uuid_key') THEN
    ALTER TABLE datasheets ADD CONSTRAINT datasheets_uuid_key UNIQUE (uuid);
  END IF;
END $$;
```

**On Postgres 11+** (we're on 18.2), `ADD COLUMN ... NOT NULL DEFAULT ...` with a non-volatile default is a metadata-only change ‚Äî no table rewrite, instant. Existing rows get UUIDs lazily via the default; no explicit backfill step needed. Fallback if a rewrite is observed: add nullable, `UPDATE ... SET uuid = gen_random_uuid() WHERE uuid IS NULL`, then `ALTER ... SET NOT NULL`.

### C. REST server ‚Äî minimal wiring

**`rest_server/asset_create_models.py`**

Add optional client-supplied UUID to the request models (matches Neo4j's `if 'id' not in model_card` behavior):

- `AssetModelCardCreate`: add `uuid: UUID | None = None`
- `AssetDatasheetCreate`: add `uuid: UUID | None = None`

**`rest_server/models.py`**

- `ModelCardDetail`: add `uuid: str` field. Keep `external_id: int` unchanged for backward compatibility.
- `DatasheetDetail` (or symmetric model): same.
- `AssetIngestResult`: add `asset_uuid: str | None = None` so POST responses surface the new UUID alongside the bigint `asset_id`.

**`rest_server/routes/assets.py`**

- `_create_model_card_in_tx` (line 220): include `uuid` in the INSERT column list, use `COALESCE($N, gen_random_uuid())` to honor a client-supplied UUID or fall back to the DB default. Add `uuid` to `RETURNING` and surface it in `AssetIngestResult.asset_uuid`.
- `_create_datasheet_in_tx` (line 403): same change for datasheets.

**`rest_server/routes/model_cards.py`** (line 396)

- Update the SELECT that builds `model_card_row` to include `uuid`.
- Pass `uuid=model_card_row["uuid"]` into `ModelCardDetail(...)`.

Symmetric change for the datasheet detail handler if one exists.

## Rollback

All changes are additive and non-destructive:

```sql
ALTER TABLE model_cards DROP COLUMN uuid;
ALTER TABLE datasheets  DROP COLUMN uuid;
-- pgcrypto can stay (harmless dependency).
```

REST-server changes revert via `git revert`. No FK references to UUID exist, so nothing else needs cleanup.

## Open items (follow-up PRs)

1. **Resolvability RFC.** Stand up a resolver at `patra.pods.icicleai.tapis.io/modelcard/<uuid>` (or under an ICICLE domain), and/or register cards with DOI / Handle / ARK. Per Beth's guardrail, the UUID isn't a true PID until this lands.
2. **Deprecate `external_id` in the API.** Once UUID is the canonical external identifier and consumers have migrated, mark `external_id: int` deprecated in the OpenAPI spec with a sunset date.
3. **Widen `_find_duplicate_model_card` key.** The Neo4j version matched on 10 fields (name, version, short_description, full_description, keywords, author, input_data, output_data, input_type, category); the Postgres version currently matches on 4 (name, version, author, short_description). Consider restoring the broader match.
4. **Frontend + MCP adoption.** Update `patra-frontend/` and `mcp_server/` to prefer `uuid` over `external_id` when constructing model-card URLs and tool responses.
5. **UUID on `models`?** Currently out of scope. Revisit if `models` becomes externally addressable (e.g. `/v1/models/{uuid}` endpoint).
